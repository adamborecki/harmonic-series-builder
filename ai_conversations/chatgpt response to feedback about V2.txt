2) The biggest “must-fix” items for the next prompt

A) Information hierarchy: what students see first

You’re 100% right: the first thing should be the current task, not audio buttons.

Next prompt change

B) Deterministic flow: NO auto-advance, ever

Next prompt change
	•	Step A always requires clicking “Check Note Position” to move to Step B.
	•	Step B always requires clicking “Check Accidental” to complete partial.

C) Fundamental: B2 confusion

You’re correct about staff reading: B2 is the 2nd line of bass clef (lines: G2–B2–D3–F3–A3).
So if it told you the octave was wrong, that’s almost certainly a bug in one of:
	•	the midi↔octave mapping ((oct+1)*12 + pc)
	•	VexFlow key string octave (b/2 means B2 in scientific pitch notation)
	•	the app starting you at an unexpected octave (e.g., C4)

Next prompt change
	•	Explicitly define octave convention: scientific pitch notation and verify: C4 = MIDI 60.
	•	Add a tiny debug line in Advanced: “Your MIDI: __, Target MIDI: __” (only in Advanced).

D) Keyboard: always-visible but not answer-revealing

You nailed the distinction: we want a learning tool, not a “show the answer” overlay.

Next prompt change
	•	Keyboard always visible, current note highlighted.
	•	Half-step check appears only in Step B and is interactive:
	•	Start (0) label on reference key, not lit.
	•	Student must click “Next half-step →” to advance (and “Prev” when needed).
	•	In Reminders ON: show numbers on the stepped keys.
	•	In Reminders OFF: no numbers.

And: require the keyboard implementation to be correct (black keys as separate overlay elements).

E) Notation spacing + ghost-note alignment

Your complaint that the treble ghost note is “to the right of bass” is a formatter/voice alignment issue. In both Gemini v2 and Claude v2, they’re building treble/bass arrays, then formatting voices—but the tickables differ (ghost notes vs real notes, different durations, etc.), so VexFlow ends up spacing them differently.    

Next prompt change (the big one)
	•	All partial notes should share the same x-position per partial across treble and bass.
	•	Easiest reliable method: for each partial index i, always add one tickable to treble voice and one tickable to bass voice at the same index:
	•	If the “home” note is treble: treble has the note, bass has a GhostNote of the same duration.
	•	If “home” is bass: bass has the note, treble has GhostNote of the same duration.
	•	If overlap window: treble has real + bass has parenthetical (or vice versa) at the same index.
	•	Use one duration type for partials 1–8 (I agree: stemless / rhythmless). Whole notes are fine.

Also: you want melodic/horizontal placement, not vertical stacking collisions. So:
	•	Use a single measure with 8 “slots” left-to-right, not a pile in one x position.

F) Horizontal scroll rule

You’re right: horizontal scroll is a “no-no” early. It should only appear when there are enough notes that you can’t reasonably compress.

Next prompt change
	•	Renderer width should fit container; format notes to fit (shrink spacing) for 1–8.
	•	Only allow horizontal scroll after completion when showing 9–16 (optional) or if viewport is extremely narrow.

G) Copy tweaks you asked for

	•	Frequency display: round to nearest integer Hz (no decimals).
